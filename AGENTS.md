# AGENTS.md

本文件定义保险产品中心后台系统的 Agent 协作规则、架构边界和工程约束。任何自动化编码、重构、评审、测试生成都必须遵循本文件。

## 项目定位

保险产品中心是保险业务中台能力，负责统一管理保险产品、责任、费率、投保规则、销售规则、上下架、版本和对外查询能力。

系统边界：
- 管理保险产品主数据、产品版本、责任配置、费率配置、销售区域、投保规则、承保规则引用、上下架状态。
- 对外提供稳定的产品查询、规则查询、版本查询接口。
- 不直接负责保单承保、支付、核保决策、理赔、佣金结算。
- 不在产品中心内实现复杂规则引擎，可引用规则编码或规则服务结果。

## 技术栈

- 后端：Java 17+、Spring Boot 3.x、Spring Cloud 2023.x 可选。
- 前端：Vue 3、TypeScript、Vite、Pinia、Vue Router。
- 数据库：MySQL 8.x。
- 缓存：Redis 7.x。
- ORM：MyBatis-Plus 或 MyBatis，禁止在复杂查询中过度依赖自动生成 SQL。
- API 文档：OpenAPI 3 / Knife4j。
- 构建：Maven 或 Gradle，后端优先 Maven。
- 测试：JUnit 5、Mockito、Testcontainers 可选。

## 核心架构

后端采用分层架构，必要时按领域拆分模块。

推荐模块：
- `product-domain`：领域模型、领域服务、领域事件、枚举和值对象。
- `product-application`：用例编排、事务边界、DTO 转换、权限校验入口。
- `product-infrastructure`：数据库、缓存、外部服务、消息、文件存储。
- `product-api`：Controller、OpenAPI、请求响应模型。
- `product-admin-ui`：Vue3 管理后台。

依赖方向：
- API 层依赖 Application。
- Application 依赖 Domain，并通过接口依赖 Infrastructure。
- Infrastructure 实现 Domain/Application 定义的仓储或外部网关接口。
- Domain 不依赖 Spring、MyBatis、Redis、HTTP、数据库实体。

## 领域设计规则

核心聚合建议：
- `InsuranceProduct`：产品主聚合，包含产品编码、名称、险种类型、公司、状态、版本信息。
- `ProductVersion`：产品版本，承载生效时间、失效时间、发布状态。
- `CoverageLiability`：保险责任，描述责任编码、责任名称、保额、免赔、等待期等配置。
- `PremiumRate`：费率配置，按年龄、性别、期间、缴费方式、保额区间等维度配置。
- `SalesRule`：销售规则，包含销售渠道、区域、职业类别、年龄范围、投保限额等。
- `UnderwritingRuleRef`：核保规则引用，只保存规则标识、版本和适用范围。

业务约束：
- 产品编码全局唯一，发布后不可修改。
- 产品发布必须绑定明确版本，版本一经生效不得直接覆盖，只能创建新版本。
- 上架产品必须通过完整性校验，包括责任、费率、销售规则、生效时间。
- 删除优先使用逻辑删除或停用，不允许物理删除已发布业务数据。
- 时间字段必须区分创建时间、更新时间、生效时间、失效时间。
- 金额使用 `BigDecimal`，禁止使用 `double` 或 `float`。
- 状态变更必须经过明确状态机，不允许直接任意更新状态字段。

## 后端编码规则

Java 规则：
- 使用 Java 17 语法，保持代码清晰，不为炫技使用复杂 Stream。
- Controller 只做参数校验、认证上下文读取和用例调用，不写业务逻辑。
- Application Service 负责事务和用例编排。
- Domain Service 负责跨实体领域规则。
- Repository 接口表达业务语义，不暴露 MyBatis Wrapper 到上层。
- DTO、Command、Query、VO 分离，不复用数据库实体作为接口入参或响应。
- 枚举必须包含稳定 code 和中文 desc，数据库保存 code。
- 统一异常模型，禁止直接向前端暴露底层异常信息。
- 所有代码方法，类，静态常量，关键点，实体类和实体类字段，数据库字段和表明都增加中文注释

接口规则：
- 后台管理接口路径使用 `/admin/api/v1/...`。
- 开放查询接口路径使用 `/open/api/v1/...`。
- 接口必须具备幂等、分页、排序、过滤条件边界。
- 分页参数统一为 `pageNo`、`pageSize`，限制最大 `pageSize`。
- 所有写接口必须记录操作人、操作时间和审计日志。

事务规则：
- 事务边界放在 Application Service。
- 禁止在循环中执行大量单条数据库写入，必须批量化或拆分任务。
- 事务内禁止调用不可靠外部系统；必要时使用事件或消息最终一致。

## 数据库规则

MySQL 规则：
- 表名使用小写下划线，例如 `product_info`。
- 主键使用 `BIGINT`，可采用雪花 ID 或数据库自增，项目内必须统一。
- 所有业务表必须包含 `id`、`created_at`、`created_by`、`updated_at`、`updated_by`、`deleted`。
- 发布态、上下架态、审核态等状态字段必须有注释和枚举映射。
- 唯一业务键必须建立唯一索引，例如产品编码、产品编码加版本号。
- 高频查询字段必须建立组合索引，索引顺序按等值、范围、排序字段设计。
- JSON 字段只用于低频扩展属性，核心查询字段必须结构化。
- 金额字段使用 `DECIMAL(18,2)` 或按业务精度调整。

迁移规则：
- 所有表结构变更必须提供迁移脚本。
- 禁止直接修改已发布迁移脚本，应追加新脚本。
- 字段新增优先兼容老数据，避免一次性强制非空导致上线失败。

## Redis 规则

缓存适用场景：
- 已发布产品详情。
- 产品版本配置快照。
- 费率和销售规则查询结果。
- 字典、枚举、配置类低频变更数据。

缓存规则：
- Key 必须带业务前缀，例如 `product:center:product:{productCode}:v:{version}`。
- 缓存必须设置 TTL，不允许永久缓存业务数据。
- 产品发布、下架、版本变更必须主动删除或刷新相关缓存。
- 防止缓存穿透，空结果可短 TTL 缓存。
- 防止缓存击穿，热点数据使用互斥锁、逻辑过期或预热策略。
- 禁止把用户敏感信息和完整内部异常写入 Redis。

## 前端规则

Vue3 规则：
- 使用 Composition API 和 TypeScript。
- 页面按业务模块组织，例如 `product`、`rate`、`liability`、`rule`、`audit`。
- API 类型必须显式声明，不允许大量 `any`。
- 表单校验前端做体验校验，后端仍必须做强校验。
- 状态枚举从统一字典或常量模块维护，避免页面硬编码。
- 列表页必须支持分页、条件查询、重置、刷新。
- 复杂编辑页拆分为基础信息、责任配置、费率配置、销售规则、发布校验。

推荐目录：
- `src/api`：接口请求。
- `src/views`：页面。
- `src/components`：业务组件和通用组件。
- `src/stores`：Pinia 状态。
- `src/router`：路由。
- `src/types`：类型定义。
- `src/utils`：工具函数。

## 安全与合规

- 所有后台接口必须接入认证和权限校验。
- 产品发布、下架、费率变更、规则变更必须记录审计日志。
- 敏感配置不得提交到代码仓库。
- 日志禁止打印身份证号、手机号、银行卡号等敏感信息；必要时脱敏。
- 导出功能必须受权限控制，并限制导出数据量。
- 管理后台操作必须具备可追溯性。

## 测试规则

必须覆盖：
- 产品创建、编辑、发布、下架状态流转。
- 产品版本生效和失效逻辑。
- 费率匹配边界条件。
- 销售规则年龄、地区、渠道、职业限制。
- 缓存刷新和缓存失效。
- 权限和审计关键路径。

测试要求：
- 领域规则优先写单元测试。
- 数据库复杂查询写集成测试。
- 前端关键表单和状态流转写组件或 E2E 测试。
- 修复 bug 必须补充回归测试。

## Agent 工作规则

在修改代码前：
- 先阅读相关模块和现有风格。
- 明确影响范围，包括数据库、缓存、接口、前端页面。
- 不得假设业务规则，遇到关键歧义应提出问题或使用最保守实现。

在实现时：
- 小步修改，保持提交可审查。
- 不引入无必要的新框架。
- 不绕过现有架构边界。
- 不把临时脚本、调试日志、测试数据提交到生产代码。
- 不使用物理删除处理已发布产品数据。

在交付前：
- 运行相关测试、格式化和静态检查。
- 说明改动范围、验证结果和未覆盖风险。
- 如果涉及数据库变更，必须说明迁移脚本和兼容策略。
