# 保险公司产品中心缺失功能详细说明书 V1.0

## 1. 文档目的

本文档用于补充当前项目在“保险公司自营产品中心”场景下的缺失能力，形成可直接拆解研发任务的功能说明。

适用对象：

- 产品经理、业务架构师
- 后端/前端开发工程师
- 测试、合规、精算、核保、运营团队

## 2. 当前基线与差距概览

### 2.1 当前已具备能力（代码已实现）

- 产品主档与版本基础流程：创建、查询、编辑、提审、审核、发布、下架
- 险种、责任、条款引用基础维护
- 发布校验与发布快照基础能力
- 管理端产品列表页面（其余模块多为占位）

### 2.2 当前核心差距

当前实现更偏“产品配置骨架”，距保险公司可生产使用的产品中心仍缺以下能力：

1. 精算与定价全流程
2. 合规报备与法务版本链路
3. 核保规则深度配置与决策联动
4. 再保与成本收益联动
5. 生命周期经营分析与闭环治理
6. 渠道差异化与机构准入控制
7. 主数据治理与跨系统一致性

## 3. 缺失模块详细说明

## 3.1 精算与定价中心

### 3.1.1 业务目标

- 建立费率从“编辑、校验、审核、发布、回溯”的完整链路
- 支持精算假设沉淀与变更留痕
- 支撑产品定价评审与调价决策

### 3.1.2 功能范围

- 费率因子管理：年龄、性别、职业、保险期间、缴费期间、保额档位、渠道
- 费率表管理：导入、模板校验、差异对比、发布与废弃
- 费率试算：单点试算、批量试算、样本回归试算
- 精算假设管理：预定利率、费用假设、发生率假设、分红演示假设
- 费率版本绑定：产品版本与费率版本强绑定

### 3.1.3 关键流程

- 精算师创建费率草稿 -> 导入并校验 -> 提交审核 -> 审核通过 -> 发布费率版本 -> 绑定产品版本
- 产品发布前必须校验“存在已发布费率版本且匹配当前方案矩阵”

### 3.1.4 数据模型建议

- `rate_factor_schema`
- `rate_table`
- `rate_table_version`
- `rate_item`
- `actuarial_assumption`
- `rate_audit_record`

### 3.1.5 接口建议

- `POST /admin/api/v1/rate-tables/import`
- `POST /admin/api/v1/rate-tables/{id}/validate`
- `POST /admin/api/v1/rate-tables/{id}/submit-review`
- `POST /admin/api/v1/rate-tables/{id}/publish`
- `POST /open/api/v1/premium/quote`

### 3.1.6 验收标准

- 费率导入失败可下载错误明细
- 未发布费率不可绑定可发布产品版本
- 同一产品版本试算结果在相同入参下可重复复现

## 3.2 合规报备与法务中心

### 3.2.1 业务目标

- 保障产品上线前具备可审计、可追溯的合规链路
- 形成条款、批文、备案材料的一致版本关系

### 3.2.2 功能范围

- 监管报备材料管理：报备包、状态、受理号、批复号
- 法务审阅：条款变更比对、审阅意见、审阅结论
- 合规规则校验：渠道合规、区域合规、文案敏感词校验
- 发布门禁：未通过合规节点不得发布

### 3.2.3 关键流程

- 产品版本提审 -> 法务审阅 -> 合规审阅 -> 监管报备 -> 批复入库 -> 允许发布

### 3.2.4 数据模型建议

- `compliance_case`
- `compliance_check_item`
- `filing_record`
- `legal_review_record`
- `document_version_link`

### 3.2.5 接口建议

- `POST /admin/api/v1/compliance/cases`
- `POST /admin/api/v1/compliance/cases/{id}/submit`
- `POST /admin/api/v1/compliance/cases/{id}/approve`
- `POST /admin/api/v1/filings`
- `GET /admin/api/v1/filings/{id}`

### 3.2.6 验收标准

- 无批文或备案受理信息时禁止发布
- 合规驳回必须带结构化原因并可追溯到字段
- 审阅记录不可物理删除

## 3.3 核保规则中心（深度版）

### 3.3.1 业务目标

- 将“规则引用”升级为“可配置、可测试、可回放”的核保规则体系
- 降低人工核保压力，提升自动核保覆盖率

### 3.3.2 功能范围

- 健康告知规则：问卷版本、跳题逻辑、异常项结论
- 职业与财务核保：职业等级限制、保额阈值、财务问卷规则
- 核保策略：自动承保、加费承保、人工核保、拒保
- 规则沙箱：规则模拟、回放与命中说明

### 3.3.3 关键流程

- 规则配置 -> 规则测试 -> 版本冻结 -> 发布绑定 -> 线上决策 -> 结果回流

### 3.3.4 数据模型建议

- `uw_rule_set`
- `uw_rule`
- `uw_rule_version`
- `uw_questionnaire`
- `uw_decision_log`

### 3.3.5 接口建议

- `POST /admin/api/v1/uw/rule-sets`
- `POST /admin/api/v1/uw/rule-sets/{id}/simulate`
- `POST /admin/api/v1/uw/rule-sets/{id}/publish`
- `POST /open/api/v1/uw/evaluate`

### 3.3.6 验收标准

- 规则模拟结果应返回命中规则链路
- 规则版本发布后不可直接修改
- 核保决策日志满足稽核追溯要求

## 3.4 再保与成本收益联动中心

### 3.4.1 业务目标

- 将产品设计与再保安排、风险暴露、成本收益统一管理
- 控制大额风险与产品利润波动

### 3.4.2 功能范围

- 再保方案：自留额、分保比例、分保方式、超赔边界
- 再保供应商准入：资质、合同有效期、可承接险种
- 成本收益测算：保费收入、手续费、赔付预期、再保成本、利润率
- 高额件风控门禁：超过阈值需特批流程

### 3.4.3 数据模型建议

- `reinsurance_scheme`
- `reinsurance_partner`
- `reinsurance_contract`
- `product_profit_projection`
- `risk_exposure_limit`

### 3.4.4 接口建议

- `POST /admin/api/v1/reinsurance/schemes`
- `POST /admin/api/v1/reinsurance/schemes/{id}/bind-product-version`
- `GET /admin/api/v1/products/{id}/profit-projection`

### 3.4.5 验收标准

- 无有效再保方案的高风险产品不能发布
- 产品发布前输出收益测算摘要并入审计日志
- 再保合同到期前自动预警

## 3.5 生命周期经营分析中心

### 3.5.1 业务目标

- 实现产品“设计-上市-经营-优化-下架”闭环
- 用经营指标驱动调价、停售和版本迭代

### 3.5.2 功能范围

- 经营指标：销量、件均保费、赔付率、继续率、退保率、利润贡献
- 预警规则：异常波动阈值、渠道偏差、地区偏差
- 运营动作：调价建议、停售建议、规则调整建议
- 复盘报告：版本表现对比、策略收益归因

### 3.5.3 数据模型建议

- `product_kpi_daily`
- `product_kpi_monthly`
- `product_alert_rule`
- `product_alert_event`
- `product_optimization_task`

### 3.5.4 接口建议

- `GET /admin/api/v1/products/{id}/kpi`
- `GET /admin/api/v1/products/{id}/alerts`
- `POST /admin/api/v1/products/{id}/optimization-tasks`

### 3.5.5 验收标准

- 指标数据支持按产品、版本、渠道、地区钻取
- 告警触发后可形成待办任务并闭环
- 关键指标历史不可被覆盖修改

## 3.6 渠道与机构差异化配置中心

### 3.6.1 业务目标

- 支持同一产品在不同渠道、机构执行差异化销售策略
- 避免“一套配置全渠道通用”带来的合规与经营风险

### 3.6.2 功能范围

- 渠道准入：APP、银保、个险、团险、经代等
- 区域准入：省市机构维度启停售
- 渠道差异规则：可售责任、费率折扣、投保限制、文案模板
- 机构授权：按分公司/营业部授权可售产品版本

### 3.6.3 数据模型建议

- `channel_profile`
- `region_profile`
- `product_channel_rule`
- `product_region_rule`
- `org_product_authorization`

### 3.6.4 接口建议

- `POST /admin/api/v1/products/{id}/channel-rules`
- `POST /admin/api/v1/products/{id}/region-rules`
- `GET /open/api/v1/products/{productCode}/sellable-scope`

### 3.6.5 验收标准

- 渠道/区域禁售规则在开放查询与试算层面均生效
- 机构未授权时不可销售和不可出单
- 规则变更具备版本与审计记录

## 3.7 主数据治理与跨系统一致性中心

### 3.7.1 业务目标

- 统一主数据编码与字典口径，减少跨系统数据偏差
- 支撑产品快照在承保、保全、理赔等系统稳定消费

### 3.7.2 功能范围

- 主数据域：险种码、责任码、疾病码、职业码、地区码、渠道码
- 字典版本管理：生效时间、停用时间、兼容策略
- 数据质量规则：完整性、一致性、唯一性、引用关系
- 同步机制：主数据发布事件、增量同步、重放

### 3.7.3 数据模型建议

- `mdm_dictionary`
- `mdm_dictionary_item`
- `mdm_version`
- `mdm_sync_task`
- `mdm_quality_issue`

### 3.7.4 接口建议

- `POST /admin/api/v1/mdm/dictionaries`
- `POST /admin/api/v1/mdm/dictionaries/{id}/publish`
- `GET /open/api/v1/mdm/dictionaries/{code}`

### 3.7.5 验收标准

- 已发布产品引用的字典项不允许物理删除
- 下游同步失败有补偿机制和告警机制
- 主数据变更可追溯到操作人和审批单

## 4. 跨模块通用能力补齐

以下能力需在所有新增模块统一补齐：

- 认证鉴权：菜单、按钮、数据权限
- 审计追踪：写操作留痕、版本留痕、审批留痕
- 状态机约束：禁止越级状态跳转
- 幂等与重试：写接口幂等、异步补偿任务
- 可观测性：指标、日志、链路追踪、告警

## 5. 分阶段落地建议

### 5.1 第一阶段（高优先级，生产门槛）

- 精算与定价中心
- 合规报备与法务中心
- 核保规则中心（基础到中级）

目标：满足“可发布、可审计、可追溯”的保险公司上线门槛。

### 5.2 第二阶段（经营与风险增强）

- 再保与成本收益联动
- 渠道与机构差异化配置

目标：实现风险可控与渠道精细化运营。

### 5.3 第三阶段（治理与优化）

- 生命周期经营分析
- 主数据治理与跨系统一致性

目标：实现产品经营闭环与集团级数据一致性。

## 6. 与现有代码结构的映射建议

- `product-domain`：补充精算、合规、核保、再保、经营分析等领域模型与状态机
- `product-application`：新增各模块用例编排服务、审批与发布门禁编排
- `product-infrastructure`：从内存实现迁移为 MySQL + Redis + MQ + 外部服务网关
- `product-api`：按 `admin/open` 分离补齐接口，保证查询与写入边界清晰
- `product-admin-ui`：将占位页替换为业务模块页面，并补齐模拟器、对比器、审核工作台

## 7. 验收基线（整体）

- 新增模块均具备：功能验收、接口验收、权限验收、审计验收、回归验收
- 所有发布门禁必须可解释：返回结构化失败原因（模块、规则、字段）
- 关键路径具备自动化测试：单元测试、集成测试、端到端测试
